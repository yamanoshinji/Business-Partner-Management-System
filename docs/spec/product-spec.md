# 協力会社員管理システム 仕様書

最終更新: 2026-03-31

---

## 1. 目的

社内管理者が、協力会社員の情報・契約期間・契約手続きの進捗を一元管理し、契約更新の見落としをなくす。

---

## 2. 現在の提供範囲

### 2.1 実装済み基盤機能

| # | 機能 | 概要 |
| --- | --- | --- |
| 1 | 企業情報 CRUD | 協力会社の基本情報を登録・編集・削除する |
| 2 | 担当者情報 CRUD | 協力会社員の基本情報を登録・編集・削除する |
| 3 | 契約管理 | 契約期間・契約手続きステータスを管理する |
| 4 | アプリ内アラート | 契約終了日のひと月前の第2月曜日を基準に通知を生成する |
| 5 | 認証 | 管理者 1 ロールのみでログイン管理する |
| 6 | 初期データ取込 | Excel ファイルから企業・担当者・契約情報を一括登録する |

### 2.2 機能追加フェーズの方針

- MVP は完了済みとし、以降は業務要件に沿った機能追加を許可する。
- 新機能は既存のドメインモデル、認証方式、契約ステータス運用と矛盾しないように設計する。
- 大きな仕様変更を伴う機能追加では、実装に先立ってこの仕様書の該当セクションを更新する。
- 次のような変更は影響が大きいため、前提条件を明文化してから実装する。
  - 認証方式や権限管理の見直し
  - 外部通知チャネルの追加
  - 外部システム連携
  - 本番運用を前提としたインフラ変更

## 5. 契約手続きステータス

### 5.1 ステータス一覧

| 値 (enum key) | 表示名 | アクター |
| --- | --- | --- |
| `NOT_STARTED` | 未着手 | - |
| `ESTIMATE_REQUESTED` | 見積依頼済み | 顧客へ依頼 |
| `ESTIMATE_RECEIVED` | 見積受領済み | 管理者が受領 |
| `DRAFT_CREATED` | 個別契約書案作成済み | 管理者が作成 |
| `DRAFT_CONFIRMED` | 個別契約書案確認依頼済み | 顧客へ確認依頼 |
| `APPROVAL_CREATED` | 稟議作成済み | 管理者が作成 |
| `CONTRACT_SENT` | 個別契約書送付済み | 経営管理部が送付 |
| `CONTRACT_SIGNED` | 個別契約書押印済み | 顧客が押印 |
| `CONTRACT_RECEIVED` | 個別契約書受領済み | 管理者が受領 |
| `SENT_TO_MANAGEMENT` | 経営管理部へ送付済み | 管理者が送付 |
| `COMPLETED` | 完了 | - |

### 5.2 ステータス遷移ルール

- `alert_date` 当日は `COMPLETED` から `NOT_STARTED` へサーバー側で自動切替する。
- 順方向は隣接する次のステータスへのみ進める。
- 差し戻しは 1 段階のみ許可する。
- `COMPLETED` からの差し戻しは許可しない。
- 上記以外の遷移は API でバリデーションエラーを返す。

## 6. アラート仕様

### 6.1 アラート日計算ルール

契約終了日 (`end_date`) に基づき、以下のロジックで `alert_date` を算出する。

1. `end_date` の 1 ヶ月前の月を求める。
2. 対象月の第 2 月曜日の日付を求める。
3. その日が日本の祝日または土日に該当する場合は翌平日にスライドする。
4. 算出結果を `contract.alert_date` に保存する。

### 6.2 祝日判定

- `holidays-jp` パッケージを使用する。
- 祝日データ取得に失敗した場合はメモリキャッシュを利用する。
- キャッシュも利用できない場合は祝日なしとして計算し、UI に警告を表示する。

### 6.3 通知トリガー

- `alert_date` が当日と一致した場合、サーバー起動時またはページアクセス時に通知を生成する。
- 同一契約の同一 `alert_date` で通知が重複しないよう、生成前に存在チェックを行う。

### 6.4 通知メッセージフォーマット

```print
「{担当者名}」({企業名}) の契約が {end_date} に終了します。手続きを確認してください。
```

## 10. 新機能追加時の完了条件

- 仕様変更がある場合、この仕様書または関連ドキュメントが更新されていること。
- 正常系と異常系のテストが追加または更新されていること。
- 既存の契約ステータス運用、通知生成、認証動作を壊していないこと。
- UI と API の両方でエラー時の扱いが明確であること。
