# 協力会社員管理システム MVP仕様書

最終更新: 2026-03-29

---

## 1. 目的

外注管理担当者（社内管理者）が、協力会社員の情報・契約期間・契約手続きの進捗を一元管理し、契約更新の見落としをなくす。

---

## 2. MVP スコープ

### 含む機能

| # | 機能 | 概要 |
|---|------|------|
| 1 | 企業情報 CRUD | 協力会社の基本情報を登録・編集・削除する |
| 2 | 担当者情報 CRUD | 協力会社員の基本情報を登録・編集・削除する |
| 3 | 契約管理 | 契約期間・契約手続きステータスを管理する |
| 4 | アプリ内アラート | 契約終了日の「ひと月前の第二月曜日」に通知を生成する |
| 5 | 認証 | 管理者 1 ロールのみ（ローカル運用前提） |

### 除外（MVP 非対象）

- メール通知
- 本番環境へのデプロイ
- 複数ロール・権限管理
- CSV 入出力
- 監査ログ

## 5. 契約手続きステータス

### 5.1 ステータス一覧

| 値 (enum key) | 表示名 | アクター |
|--------------|--------|---------|
| `NOT_STARTED` | 未着手 | — |
| `ESTIMATE_REQUESTED` | 見積依頼済み | 顧客（協力会社）へ依頼 |
| `ESTIMATE_RECEIVED` | 見積受領済み | 自分（管理者）が受領 |
| `DRAFT_CREATED` | 個別契約書案作成済み | 自分が作成 |
| `DRAFT_CONFIRMED` | 個別契約書案確認依頼済み | 顧客へ確認依頼 |
| `APPROVAL_CREATED` | 稟議作成済み | 自分が作成 |
| `CONTRACT_SENT` | 個別契約書送付済み | 経営管理部が送付 |
| `CONTRACT_SIGNED` | 個別契約書押印済み | 顧客が押印 |
| `CONTRACT_RECEIVED` | 個別契約書受領済み | 自分が受領 |
| `SENT_TO_MANAGEMENT` | 経営管理部へ送付済み | 自分が送付 |
| `COMPLETED` | 完了 | — |

### 5.2 ステータス遷移ルール

**ステータス更新**: `alert_date` になったら、`COMPLETED` から `NOT_STARTED` へ自動切替

**前進（順方向）**: 隣接する次のステータスへのみ進める（スキップ不可）。

**差し戻し（後退）**: **1 段階のみ**後退可能とする。それより前は許可しない。
> 理由: 複数段階の差し戻しは実運用で稀であり、誤操作リスクの方が高いため。

**例外**:
- `COMPLETED` からの差し戻しは不可（変更は note への記録で対応）。

```
NOT_STARTED
  ↓ ↑(1段階差し戻し)
ESTIMATE_REQUESTED
  ↓ ↑
ESTIMATE_RECEIVED
  ↓ ↑
DRAFT_CREATED
  ↓ ↑
DRAFT_CONFIRMED
  ↓ ↑
APPROVAL_CREATED
  ↓ ↑
CONTRACT_SENT
  ↓ ↑
CONTRACT_SIGNED
  ↓ ↑
CONTRACT_RECEIVED
  ↓ ↑
SENT_TO_MANAGEMENT
  ↓
COMPLETED（差し戻し不可）
```

## 6. アラート仕様

### 6.1 アラート日計算ルール

契約終了日 (`end_date`) に基づき、以下のロジックでアラート日を算出する。

1. `end_date` の **1 ヶ月前の月** を求める。
   - 例: end_date = 2026-04-30 → 対象月 = 2026-03
2. 対象月の **第 2 月曜日** の日付を求める。
3. その日が **日本の祝日・土日** に該当する場合、**翌平日** にスライドする。
4. 算出結果を `contract.alert_date` に保存する。

### 6.2 祝日判定

- 日本の祝日 API（`holidays-jp` npm パッケージ、または `https://holidays-jp.github.io/api/v1/` 等）を使用する。
- **API 障害・オフライン時のフォールバック**: アプリ起動時に取得した祝日データをメモリキャッシュし、キャッシュが存在する場合はそちらを使う。キャッシュも存在しない場合は「祝日なし」として計算し、画面に「祝日情報を取得できませんでした」と表示する。

### 6.3 通知トリガー

- `alert_date` が当日と一致した場合、サーバー起動時またはページアクセス時に通知を生成する（MVP はポーリング方式）。
- 同一契約の同一 `alert_date` で通知が重複しないよう、生成前に存在チェックを行う。

### 6.4 通知メッセージフォーマット

```
「{担当者名}」({企業名}) の契約が {end_date} に終了します。手続きを確認してください。
```

## 10. 受け入れ条件（MVP 完了の定義）

- [ ] 企業・担当者の作成・編集・削除が正常に動作する
- [ ] 契約を登録し、ステータスを順方向に 1 段階進められる
- [ ] ステータスを 1 段階差し戻しできる（COMPLETED からは不可）
- [ ] 契約終了日を設定すると alert_date が自動計算される
- [ ] alert_date 当日にアプリ内通知が生成される（重複なし）
- [ ] 祝日 API 障害時にフォールバックが動作する
- [ ] ログイン認証が機能し、未認証時はログイン画面にリダイレクトされる
